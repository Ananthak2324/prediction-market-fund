1. WHAT THIS PROJECT DOES
================================================================================

EdgeFund is a paper-trading system that exploits pricing gaps between:
  - Kalshi prediction markets  (retail-driven prices, behavioral biases)
  - Pinnacle sportsbook         (sharp market, vig-free implied probabilities)

The thesis: Kalshi retail bettors systematically misprice game contracts
relative to Pinnacle's sharp consensus. When the gap between Kalshi's
probability and Pinnacle's vig-free probability exceeds a threshold,
we paper-trade the discrepancy and log outcomes to build a statistical
track record.

Example:
  Marlins win probability
    Kalshi: 49%
    Pinnacle (vig-free): 61%
    Gap: -12.2% (Kalshi is underpricing the home team)
  → Signal: BUY_YES on KXMLBGAME-26JUN241210TEXMIA-MIA at Kalshi

The system is NOT live trading real money. It is building a statistical
track record to demonstrate edge before applying for YC (deadline July 27, 2026)
and eventually deploying capital.

7. DATA FLOW — HOW EVERYTHING CONNECTS
================================================================================

Every 10 minutes (LaunchAgent):
  schedule_snapshots.py
    → Polls Kalshi API for upcoming MLB/NBA games
    → For any game whose start is 110-130 min away (the 20-min window):
         Fetches live Kalshi bid/ask prices
         Fetches live Pinnacle moneyline from TheOddsAPI
         Computes gap = k_prob - v_prob
         Saves snapshot to data/snapshots/YYYY-MM-DD_HHMM.json
         Appends to data/snapshots/master_log.json
    → Games whose window was missed → data/snapshots/missed_snapshots.json

Every 2 AM daily (LaunchAgent):
  update_outcomes.py
    → Reads all snapshot files from data/snapshots/
    → For each game row flagged as tradeable (abs_gap >= 0.05):
         If trade not yet in paper_trades.json:
           → Calls research_agent.run() [AI BARRIER]
                If SKIP  → writes to data/skipped_trades.json, stops
                If TRADE or MONITOR → logs to data/paper_trades.json
         If trade already in paper_trades.json but outcome is null:
           → Checks if game started >0.5h ago
           → Fetches Kalshi settled result via API
           → Updates outcome (WIN/LOSS), correct, resolution_price, resolved_at
    → Rebuilds data/performance_summary.json with all stats + agent_stats

On demand:
  python scripts/update_outcomes.py     ← run manually anytime
  python scripts/snapshot_gaps.py       ← take a manual snapshot
  python scripts/test_agent.py          ← test agent on recent trades
  streamlit run dashboard/app.py        ← launch the dashboard

AI Agent barrier (inside update_outcomes.py):
  For EVERY new trade candidate before it's logged:
    research_agent.run(game_dict)
      → Check 1: Is Pinnacle line stable? (re-fetch from TheOddsAPI, compare to snapshot)
                  If abs(current - snapshot) >= 0.03 → SKIP immediately
      → Check 2: Web search for breaking news
                  One query: "{home} {away} starting pitcher lineup injury scratch {date}"
                  Claude claude-sonnet-4-6 + web_search_20250305 beta
      → Check 3: Synthesize verdict as JSON
    Returns: { recommendation, confidence, reasoning, gap_type, news_found, ... }
    TRADE/MONITOR → trade is logged to paper_trades.json with agent fields
    SKIP          → trade goes to skipped_trades.json only

────────────────────────────────────────
scripts/schedule_snapshots.py
────────────────────────────────────────
PURPOSE: The core data collection engine. Runs every 10 minutes via LaunchAgent.

WHAT IT DOES:
  1. Calls Kalshi /markets?series_ticker=KXMLBGAME (and KXNBAGAME) to get
     all active games with their start times and status.
  2. For each game, calculates how many minutes until first pitch.
  3. If the game is 110-130 minutes away (the 20-min snapshot window):
       a. Fetches live Kalshi ask/bid prices for both YES and NO contracts
       b. Computes mid-price (bid + ask) / 2 as k_prob
       c. Fetches Pinnacle moneyline from TheOddsAPI for that game
       d. Strips the vig using remove_vig() from core/utils.py
       e. Computes gap = k_prob - v_prob
       f. Flags if abs_gap > threshold
       g. Saves the full snapshot to data/snapshots/YYYY-MM-DD_HHMM.json
  4. If a game's window passed without being captured → logs to missed_snapshots.json
  5. Writes status to scheduler_log.txt every run

WHY 2 HOURS: The 2-hour pre-game window is when:
  - Starting lineups are confirmed but still adjustable
  - Sharp money has moved in but retail hasn't fully caught up
  - Enough time remains for the gap not to disappear immediately

KEY FUNCTION: get_well_captured_tickers()
  - Reads existing snapshot files to avoid re-snapping games already captured
  - BUG FIXED June 24: Was globbing *.json and hitting missed_snapshots.json (a list)
    Fixed to glob only YYYY-MM-DD_HHMM.json pattern

SNAPSHOT FILE FORMAT:
  {
    "snapshot_time": "2026-06-23_2338",     ← UTC, matches filename
    "rows": [
      {
        "sport": "MLB",
        "game": "Chicago Cubs @ New York Mets",
        "team": "New York Mets",             ← the side being priced
        "side": "HOME",
        "start_utc": "2026-06-26T02:10:00Z",
        "kalshi_ticker": "KXMLBGAME-...-NYM",
        "event_ticker": "KXMLBGAME-...",
        "k_prob": 0.505,                     ← Kalshi mid-price
        "k_bid": 0.50,
        "k_ask": 0.51,
        "v_prob": 0.5087,                    ← Pinnacle vig-free prob
        "pinnacle_price": -114,              ← American moneyline
        "gap": -0.0037,                      ← k_prob - v_prob
        "abs_gap": 0.0037,
        "fav_flag": false,                   ← true if abs_gap > threshold
        "result": null
      }
    ]
  }

────────────────────────────────────────
scripts/update_outcomes.py
────────────────────────────────────────
PURPOSE: Nightly script that (1) ingests new trades from snapshots,
         (2) resolves settled trades, (3) rebuilds performance_summary.json

FLOW IN DETAIL:

  ingest_new_trades():
    - Reads all snapshot files
    - Finds rows where fav_flag=True and abs_gap >= GAP_THRESHOLD (0.05)
    - For each new game not yet in paper_trades.json:
        If _AGENT_AVAILABLE:
          verdict = research_agent.run(game_dict)
          If SKIP → write to skipped_trades.json, skip logging
          Else → attach agent fields to trade dict, log to paper_trades.json
        Else:
          Log trade with agent_verdict=None
    - Returns (trades, new_count, replaced_count, skipped_count)

  resolve_trades():
    - For each trade where outcome is None and game started > RESOLVE_AFTER hours ago:
        GET /markets/{kalshi_ticker}
        Check settled_price → outcome = WIN if settled_price == 1.0 else LOSS

  build_summary():
    - Computes overall/tier win rates, gap bucket breakdown, p-value (binomtest)
    - Categorizes trades as "clean" (hours_before_game <= 3) vs "suspect" (> 3)
    - Calculates agent_stats section
    - Saves to performance_summary.json

TIMING CATEGORIES:
  - Clean trades: snapshot taken <= 3 hours before game start
    These are the trades that count for the statistical track record
  - Timing-suspect: snapshot taken > 3 hours early (may have been superseded
    by roster changes, injury news before the window)
  - Tier 1: abs_gap >= 10% (high conviction)
  - Tier 2: abs_gap 5–10% (moderate conviction)

REPLACEMENT LOGIC:
  If the same event_ticker shows up in a later snapshot (closer to game time),
  the later snapshot replaces the earlier one in paper_trades.json.
  This ensures we always have the freshest pre-game price, not an early one.

────────────────────────────────────────
agent/research_agent.py
────────────────────────────────────────
PURPOSE: AI barrier between gap detection and trade logging.
         Evaluates whether a gap is safe to trade or should be skipped.

ARCHITECTURE:
  This uses Claude claude-sonnet-4-6 with the web_search_20250305 beta tool.
  The web search runs SERVER-SIDE inside Claude — there are no client-side
  tool_use blocks. stop_reason = "end_turn" directly after Claude searches
  and synthesizes. The search counter in our cost log shows 0 but searches
  DO run; cost is absorbed into the token billing.

INPUT (game_dict):
  {
    "home_team": "Miami Marlins",
    "away_team": "Texas Rangers",
    "sport": "MLB",                          ← MLB or NBA
    "side": "HOME",                          ← which side we'd trade
    "game_time": "2026-06-24T19:10:00Z",
    "snapshot_time": "2026-06-24T17:08:00Z",
    "date": "2026-06-24",
    "kalshi_prob": 0.295,
    "pinnacle_prob": 0.55,
    "gap": -0.255,
    "gap_direction": "kalshi_lower",         ← or "kalshi_higher"
    "tier": 1,                               ← 1 if abs_gap >= 0.10, else 2
    "hours_before_game": 2.03,
    "timing_suspect": false
  }

OUTPUT (verdict dict):
  {
    "recommendation": "SKIP",               ← TRADE / SKIP / MONITOR
    "confidence": "HIGH",                   ← HIGH / MEDIUM / LOW
    "reasoning": "...",                     ← 1-2 sentence explanation
    "gap_explanation": "...",               ← why the gap exists
    "gap_type": "INFORMATIONAL",            ← BEHAVIORAL or INFORMATIONAL
    "news_found": true,
    "news_detail": "...",                   ← what news was found
    "news_source": "...",                   ← URL or outlet name
    "pitcher_confirmed": true,
    "weather_issue": false,
    "pinnacle_stable": true,
    "pinnacle_movement": 0.01               ← abs change since snapshot
  }

THREE CHECKS:
  Check 1 — Pinnacle stability (before API call)
    Re-fetches current Pinnacle line via TheOddsAPI
    If abs(current - snapshot) >= 0.03 (3pp): auto SKIP
    Sharp money moved = informational gap

  Check 2 — Web search (inside Claude API call)
    Single combined query per trade to minimize cost
    Query: "{home} {away} starting pitcher lineup injury scratch {date}"
    Claude searches and synthesizes findings

  Check 3 — Verdict synthesis
    Claude returns JSON verdict based on BEHAVIORAL vs INFORMATIONAL framework
    Behavioral gaps → TRADE or MONITOR
    Informational gaps → SKIP

DECISION RULES (from system prompt):
  SKIP if: pitcher scratched, key player injured, Pinnacle moved 3pp+,
           wind 15+ mph blowing in, bullpen overused, any credible injury
  TRADE if: roster confirmed, pitcher healthy, line stable 3h+,
            heavily favored team (favorite-longshot bias), no injury news,
            public narrative favors underdog

COST OPTIMIZATIONS:
  - System prompt uses cache_control=ephemeral → 90% cost reduction after run 1
    (5-minute cache TTL; LaunchAgent reuse within same run hits cache)
  - Single web search per trade (was 4 separate queries)
  - Search results truncated to 1500 chars
  - Actual cost: ~$0.010–$0.014 per trade (budget was $0.05)
  - Cost log: data/agent_cost_log.csv

FALLBACK: If Anthropic API fails (no key, network error, any exception):
  Returns {"recommendation": "MONITOR", "_agent_error": True, ...}
  MONITOR means the trade IS logged (conservatively). Agent failures
  never block trade logging.

────────────────────────────────────────
core/utils.py
────────────────────────────────────────
PURPOSE: Pure utility functions shared across the project.

KEY FUNCTIONS:

  american_to_prob(odds: int) -> float
    Converts American moneyline to raw implied probability.
    Positive (underdog): 100 / (odds + 100)
    Negative (favorite): abs(odds) / (abs(odds) + 100)

  remove_vig(prob_a: float, prob_b: float) -> tuple[float, float]
    Removes the bookmaker vig from a two-outcome market.
    Divides each raw prob by their sum.
    e.g. (-150, +130) → raw (0.600, 0.435) → vig-free (0.579, 0.421)

  This is the critical math that makes Pinnacle prices comparable to Kalshi.
  Kalshi prices are already vig-free (market maker sets odds, fees separate).
  Pinnacle embeds ~2-3% vig which must be removed before comparing.

  Decoding: KX = Kalshi Exchange, MLBGAME = sport,
            26JUN24 = June 24 2026, 1210 = 12:10 PM ET start,

  Date format: DDMMMYY (24 June 2026 → 24JUN26)

13. RESEARCH AGENT — ARCHITECTURE DEEP DIVE
================================================================================

The agent is called from update_outcomes.py for EVERY new trade candidate.
It acts as a barrier: only TRADE or MONITOR verdicts reach paper_trades.json.

INFORMATION FLOW:
  update_outcomes.py
    → _build_agent_game_dict(trade)          ← formats trade as game_dict
    → research_agent.run(game_dict)           ← the barrier call
         → _fetch_current_pinnacle()          ← Check 1: re-fetch Pinnacle now
         → _build_user_message()              ← builds prompt with context
         → _make_api_call()                   ← Claude with web search
         → parse JSON from response
         → _print_and_log_cost()
         → return verdict dict
    → If SKIP: _append_skipped(), continue
    → Else: trade.update(_agent_fields(verdict)), existing.append(trade)

PROMPT DESIGN:
  System prompt: ~1,500 tokens. Cached after first call.
    - Role: "sports prediction market research analyst"
    - SKIP signals: pitcher scratched, key injury, Pinnacle moved 3pp+, weather
    - TRADE signals: roster confirmed, line stable, behavioral bias patterns
    - "One confirmed injury overrides any number of behavioral signals"
    - Output: JSON only, no markdown

  User message: Built fresh per trade.
    - Game details, gap amount, tier, hours before game
    - Pinnacle stability result from Check 1
    - Warning if hours_before_game > 3.0: "EARLY SNAPSHOT — extra scrutiny"
    - Warning if abs_gap >= 0.20: "UNUSUALLY LARGE — extra scrutiny"
    - Single web search instruction with pre-built query

AGENTIC LOOP (max 8 turns):
  Exists as infrastructure but rarely fires because web_search_20250305
  runs server-side. For standard web searches, Claude returns end_turn
  directly. The loop would handle tool_use blocks from other tool types.

ERROR HANDLING:
  Any exception → returns _default_verdict() with recommendation=MONITOR
  MONITOR = trade gets logged with the uncertainty noted
  This ensures the agent is never a point of failure that blocks data collection

17. KNOWN ISSUES & EDGE CASES
================================================================================

FIXED (June 24, 2026):
  Bug: schedule_snapshots.py crashed with AttributeError: 'list' object
       has no attribute 'get'
  Cause: get_well_captured_tickers() globbed *.json and hit missed_snapshots.json
         (which is a list, not a dict)
  Fix: Changed glob pattern to ????-??-??_????.json

KNOWN QUIRKS:

  1. UTC vs CT everywhere
     All snapshot filenames are UTC. All displayed times are CT.
     The fmt_snap_time() function converts between them.
     ET = ZoneInfo("America/Chicago") in dashboard/app.py — change if needed.
     The "today_str" for filtering uses CT. This means a snapshot taken at
     11 PM CT might have a UTC filename of the next day — handled in the
     dashboard by the 24-hour lookback window.

  2. Timing-suspect trades (the 9 open ones)
     Early snapshots (hours_before_game > 3) get marked timing_suspect=True.
     These will be REPLACED when a clean snapshot (1h50m-2h10m) fires.
     The replacement logic in update_outcomes.py uses event_ticker as key.
     Until replaced, they sit in paper_trades.json as open timing-suspect trades.

  3. Agent fields missing on 14 existing trades
     All 14 trades in paper_trades.json predate the agent (logged before
     agent was wired in June 24). Their agent_verdict fields are null.
     The dashboard shows a note about this. agent_stats will stay at zero
     until post-agent trades resolve.

  4. Kalshi API pagination
     /markets returns up to 200 results by default. If a sport has more than
     200 active markets, results are paginated. The current code does not
     handle pagination. For MLB mid-season this is fine; near season start
     with all future games loaded it might truncate.

  5. TheOddsAPI game matching
     Matching Kalshi game events to TheOddsAPI events is done by time window
     (±2 hours) + team name normalization. If both teams are new to normalizer.py
     the match will fail silently and the snapshot won't have Pinnacle data.
     Fix: add missing team names to analysis/normalizer.py maps.

  6. Web search reliability
     The web_search_20250305 beta is a research preview. It can occasionally
     return outdated cached web pages. The agent is designed to be conservative
     (SKIP beats TRADE when uncertain) so false positives on "no news" are less
     damaging than false negatives on "injury detected."

  7. No NBA games (as of June 24, 2026)
     NBA Finals ended. No NBA games until October. The system handles this
     gracefully (0 games found). All current activity is MLB.
